SQL? Or noSQL? That Is the Question.
Because the first Database framework I learned was noSQL/ Document-oriented, I didn't have the context to understand why it was so innovative. The convenience of having a schema-less data store and ease of structural data-object changes was lost to me. The efficient map-reduce functionality among the other scaling benefits didn't apply to this small, demo-oriented app. For those reasons I think that MongoDB is a comfortable start to learning how to develop a database - if you are strong with OOP. Otherwise, simpler and more practical tools are available for beginers, such as Firebase which has the added convenience of providing hosting, authentication, and analytics - not to sound like an advertisement. I think it would be more prudent to either lean all in to convenience and utilize a tool like Firebase, or learn SQL/MySQL, a very popular and well-established language/management tool.

 

Determining project scale at the beginning.
The nuances of learning one Database architecture over another paled in comparison to this very critical lesson this project did teach me. My Issues with this project reflected a problem I very well knew how to rectify but never implemented strategies to circumvent. Not until I actually had to work for a client and I had to learn quickly. All of that to say that, at the time of this 3-week project, I spent the first 2 weeks building the landing page in a Javascript/JSX library I had just learned (React) and didn't implement aaaany features until the final week. No database. No Api. No Object structure. My reasons for neglecting the premise of the project were simply that I loved learning React and I loved that front-end development was limited only to what I could imagine. I imagined quite a lot.
As I retore this app, it is very apparent that I neglected to build a sustainable DB. The Object structure doesn't make sense. I have hard copies of images, which is fine but for such a small project I could also demo my aptitude and use an API to retrieve images instead of storing them locally. And to top everything off the backend did not outlast the many version updates that Node has had in the last 3 years. Considering how small an app this is, I feel that the now dead backend is in such a state more due to my negligence than version updates killing it. To rectify most of these issues I will be building the backend using Firebase, which may also eliminate the need for Express. We'll see. And I will be using the Unsplash API for some stock photos to serve as placeholders of the e-commerce shop items.
